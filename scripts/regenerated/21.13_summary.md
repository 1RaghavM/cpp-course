When a class manages a dynamically allocated resource (a pointer obtained via `new`), the compiler-generated copy constructor and copy assignment operator create problems. They perform a **shallow copy** -- they copy the pointer value itself, not the data it points to. Two objects then point to the same memory, and when both destructors run, that memory is `delete`-ed twice, causing undefined behavior.

```cpp
#include <iostream>

class IntHolder {
    int* m_ptr;
public:
    IntHolder(int v) : m_ptr{ new int{ v } } {}
    ~IntHolder() { delete m_ptr; }
    int value() const { return *m_ptr; }
    void setValue(int v) { *m_ptr = v; }
};

int main() {
    IntHolder a{ 5 };
    IntHolder b{ a }; // shallow copy: b.m_ptr == a.m_ptr
    a.setValue(99);
    std::cout << b.value(); // prints 99, not 5!
} // crash: double delete
```

The fix is a **deep copy** -- allocate fresh memory in the copy and duplicate the pointed-to data. You implement this in two places: the **copy constructor** (used when a new object is initialized from an existing one) and the **copy assignment operator** (used when an already-constructed object is assigned a new value). The assignment operator must additionally handle self-assignment (`if (this == &src) return *this;`) and release its old resource before allocating new memory.

This leads to the **Rule of Three**: if a class needs a custom destructor, it almost certainly needs a custom copy constructor and a custom copy assignment operator as well. The compiler-generated versions do shallow copies, which are only safe when the class holds no owning pointers. Whenever you write a destructor that calls `delete`, immediately write the other two.

A practical checklist for a correct deep-copy assignment operator: (1) guard against self-assignment, (2) `delete` the existing resource, (3) copy the length or size, (4) allocate new memory, (5) copy each element, (6) return `*this`. Forgetting any step leads to leaks, crashes, or corruption.
