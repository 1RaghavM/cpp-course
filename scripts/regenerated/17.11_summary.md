## C-style String Symbolic Constants

In earlier lessons you used C-style strings as null-terminated `char` arrays. There is a second, more common way C-style strings appear in C++ code: as **string literals**, which are C-style string symbolic constants.

### String literals are const char arrays

When you write `"hello"`, the compiler creates a `const char[6]` (five characters plus the null terminator `'\0'`) in a read-only section of the program's memory. This array persists for the entire lifetime of the program — it has **static storage duration**. Attempting to modify a character through a pointer to that memory is undefined behavior.

### Pointing to a string literal

Because arrays decay to pointers, a string literal decays to a `const char*`. You should always use `const char*` (never `char*`) when pointing to a string literal:

```cpp
#include <iostream>

int main()
{
    const char* greeting{ "hello" };
    std::cout << greeting << '\n';  // prints: hello

    greeting = "world";             // OK — pointer now aims at a different literal
    std::cout << greeting << '\n';  // prints: world
}
```

The pointer itself is not const, so you can reassign it to point at a different string literal. But the characters it points to are const and must not be modified.

### const char* vs std::string_view

`std::string_view` (Ch 5) is almost always a better choice than `const char*` for read-only string parameters. A `string_view` stores both a pointer and a length, so it does not need to scan for a null terminator and it can refer to substrings. Prefer `std::string_view` in new code; you will encounter `const char*` mainly when interfacing with C libraries or legacy APIs.

### Arrays of string literals

A useful pattern is an array of `const char*`, where each element points to a different string literal. This gives you a lightweight lookup table with no heap allocation:

```cpp
const char* colors[]{ "red", "green", "blue" };
```

Each element is a pointer into the read-only data segment. The array itself lives on the stack, but the strings it points to live for the entire program.
