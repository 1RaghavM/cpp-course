## Chapter 25 Summary

This chapter covered virtual functions and the runtime polymorphism system that makes inheritance powerful in C++.

### Base pointers and the wrong version (25.1)

Calling a member function through a `Base*` or `Base&` resolves based on the static type, not the actual object type. The compiler picks the base version at compile time, ignoring any derived override.

### Virtual functions and polymorphism (25.2)

Marking a base function `virtual` defers resolution to runtime. The actual object type determines which version runs, enabling polymorphism through a single base reference.

### override, final, covariant returns (25.3)

`override` makes the compiler verify the function actually overrides a base virtual, catching signature mismatches. `final` prevents further overriding or inheritance. Covariant return types let an override return a more-derived pointer or reference.

### Virtual destructors (25.4)

Deleting a derived object through a base pointer without a virtual destructor skips the derived destructor, leaking resources. Any polymorphic base class needs a virtual destructor.

### Early vs late binding (25.5)

Early binding resolves calls at compile time (fast, inflexible). Late binding defers to runtime via virtual dispatch, adding one indirection per call.

### The virtual table (25.6)

Each polymorphic class has a vtable -- a static array of function pointers. Each object stores a hidden vptr pointing to its class's vtable. Virtual calls go through vptr to vtable to function pointer. Cost: one pointer per object, one indirection per call.

### Pure virtuals, abstract classes, interfaces (25.7)

A pure virtual function (`= 0`) forces derived classes to provide an implementation. A class with any pure virtual is abstract and cannot be instantiated. An interface class has only pure virtuals and a virtual destructor.

### Virtual base classes (25.8)

The diamond problem gives a class two copies of a shared ancestor. Declaring the shared base `virtual` in intermediate classes ensures one copy. The most-derived class constructs the virtual base.

### Object slicing (25.9)

Assigning a derived object to a base variable by value copies only the base portion, losing derived data and polymorphic behavior. Use pointers or references to avoid slicing.

### Dynamic casting (25.10)

`dynamic_cast<Derived*>(basePtr)` safely checks the runtime type, returning `nullptr` on failure (or throwing `std::bad_cast` for references). Requires at least one virtual function.

### Printing inherited classes (25.11)

Since `operator<<` cannot be virtual, the pattern is to have it call a virtual `print(std::ostream&)` member. Each derived class overrides `print`, and the single `operator<<` dispatches polymorphically.
