Every expression in C++ is either an **l-value** or an **r-value**. An l-value refers to an object that persists beyond the expression that uses it -- it has an identifiable memory address. Variables, array elements, and dereferenced pointers are l-values. An r-value is a temporary that exists only within the expression that creates it. Literals (other than string literals), the return value of a function that returns by value, and the result of arithmetic operators are all r-values.

An **l-value reference** (declared with `&`) can bind to an l-value. A `const` l-value reference can also bind to an r-value -- this is why `const std::string&` can accept a temporary string. But a non-const l-value reference cannot bind to an r-value, because modifying a value that is about to be destroyed makes no sense in that context.

C++11 introduced **r-value references**, declared with `&&`. An r-value reference can bind to an r-value but not to an l-value. This gives you a way to write functions that specifically accept temporaries.

When an r-value reference binds to a temporary, **the lifetime of that temporary is extended** to match the lifetime of the reference. This means you can safely use the value for as long as the reference exists.

```cpp
#include <iostream>

void printRef(int& x)       { std::cout << "l-value: " << x << '\n'; }
void printRef(int&& x)      { std::cout << "r-value: " << x << '\n'; }

int main()
{
    int a{5};
    printRef(a);       // calls int& overload
    printRef(3 + 4);   // calls int&& overload

    int&& rref{10};    // r-value ref extends lifetime of temporary
    std::cout << rref << '\n'; // 10 -- still valid
}
```

R-value references are the key building block for move semantics. Because an r-value reference tells you the bound object is a temporary that nobody else needs, a function receiving one can safely "steal" the object's resources instead of copying them. The next lessons will show exactly how move constructors and move assignment operators use this capability. For now, the important takeaway is: `&&` identifies temporaries, and the compiler uses overload resolution to pick the `&&` overload when the argument is an r-value.
