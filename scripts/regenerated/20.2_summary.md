Every running program uses two main memory regions for storing data: the **stack** and the **heap**. Understanding how each works is essential for writing efficient, bug-free C++.

## The call stack

When you call a function, the compiler pushes a **stack frame** onto the call stack. That frame holds the function's local variables, parameters, and the return address. When the function returns, its frame is popped off and the memory is instantly reclaimed. This push/pop pattern makes stack allocation extremely fast — it only requires adjusting a single pointer.

Stack memory is limited (typically 1–8 MB). If you exhaust it — for example, by declaring a massive local array — you get a **stack overflow**, which crashes your program.

## The heap

Heap memory (also called the free store) is allocated at runtime with `new` and released with `delete`. It is much larger than the stack, but allocation is slower because the system must search for a suitable block of free memory. Heap-allocated objects persist until you explicitly free them, which means their lifetime is not tied to any particular function scope.

Failing to `delete` heap memory causes a **memory leak**; using a pointer after `delete` causes **undefined behavior**.

## Stack vs heap at a glance

- **Speed:** Stack allocation is faster (pointer bump vs. free-list search).
- **Size:** The stack is small and fixed; the heap is large.
- **Lifetime:** Stack variables die when the function returns; heap variables live until you `delete` them.
- **Management:** The compiler manages the stack automatically; you manage the heap manually (or via smart pointers, covered later).

```cpp
#include <iostream>

void showStackAddress() {
    int local{42};                    // stack
    std::cout << "Stack: " << &local << '\n';
}

int main() {
    showStackAddress();
    int* heapVal{new int{99}};        // heap
    std::cout << "Heap:  " << heapVal << '\n';
    delete heapVal;
    return 0;
}
```

Prefer stack allocation by default. Reach for the heap only when you need an object to outlive its creating scope or when the data is too large for the stack.
