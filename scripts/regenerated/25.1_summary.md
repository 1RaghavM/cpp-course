When one class inherits from another, the derived object contains a complete copy of the base class inside it. Because of this, C++ allows a `Base*` pointer or a `Base&` reference to point at or bind to a `Derived` object. This is fundamental: anywhere the program expects a `Base`, you can hand it a `Derived`.

This ability is immediately useful. You can write a single function that accepts a `Base&` and pass it objects of any derived type -- no overloading required for each subclass. You can store mixed derived types in an array of `Base*`. The compiler is satisfied because every `Derived` **is-a** `Base`.

However, there is a critical limitation. When you call a member function through a `Base*` or `Base&`, the compiler resolves the call based on the **static type** of the pointer or reference, not the actual runtime type of the object. If both `Base` and `Derived` define a function with the same name and signature, the `Base` version is called -- even though the object is really a `Derived`.

```cpp
#include <iostream>
#include <string>

class Animal {
    std::string m_name;
public:
    Animal(const std::string& name) : m_name{name} {}
    std::string_view getName() const { return m_name; }
    std::string_view speak() const { return "..."; }
};

class Cat : public Animal {
public:
    Cat(const std::string& name) : Animal{name} {}
    std::string_view speak() const { return "Meow"; }
};
```

If you pass a `Cat` to a function taking `const Animal&` and call `speak()`, you get `"..."` instead of `"Meow"`. The reference's type is `Animal`, so `Animal::speak()` is called. The `Cat::speak()` function is completely invisible to the compiler in this context.

This is not a bug -- it is how non-virtual function dispatch works. The compiler sees only the declared type of the pointer or reference and picks the corresponding function at compile time. The result is that you can **store** derived objects through base pointers, but you **cannot yet call derived-specific behavior** through them. This motivates the need for a mechanism that defers the function choice to runtime, which is the topic of the next lesson.
