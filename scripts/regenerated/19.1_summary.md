So far, every variable you have created lives on the **stack** -- its lifetime is tied to the scope where it is defined, and the compiler handles cleanup automatically. **Dynamic memory allocation** lets you request memory from the **heap** (also called the free store) at runtime, giving you explicit control over when that memory is created and destroyed.

Use the `new` keyword to allocate a single object on the heap. It returns a pointer to the newly created memory. Use `delete` to release that memory when you are done with it.

```cpp
#include <iostream>

int main()
{
    int* ptr{ new int{ 42 } }; // allocate an int on the heap
    std::cout << *ptr << '\n'; // prints 42
    delete ptr;                // free the heap memory
    ptr = nullptr;             // prevent dangling pointer
    return 0;
}
```

Three critical mistakes to avoid:

- **Memory leak:** If you lose the pointer to heap memory (e.g., the pointer goes out of scope or gets reassigned) without calling `delete`, that memory is never freed. The OS reclaims it when the program exits, but long-running programs will consume more and more memory.
- **Dangling pointer:** After `delete`, the pointer still holds the old address. Dereferencing it is undefined behavior. Always set the pointer to `nullptr` immediately after `delete`.
- **Double delete:** Calling `delete` on the same non-null pointer twice is undefined behavior. Setting the pointer to `nullptr` after delete also guards against this, because `delete nullptr` is a safe no-op.

When should you use dynamic allocation? When the size or number of objects is not known at compile time, or when an object must outlive the scope that created it. Stack memory is faster and safer by default -- prefer it unless you have a concrete reason to allocate on the heap.

`new` can fail if the system runs out of memory. By default it throws a `std::bad_alloc` exception. For now, assume allocations succeed in small programs.
