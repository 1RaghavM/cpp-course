The previous lessons introduced composition ("part-of"), aggregation ("has-a" with shared lifetime), and association (a stored pointer or reference to an unrelated object). **Dependency** is the weakest relationship: one class *uses* another temporarily without storing it as a member.

A dependency exists when an object appears only as a function parameter, a local variable, or a return type. The dependent object's scope is limited to that single function call -- once the function returns, the relationship is gone. Compare this with association, where a member variable keeps the link alive across multiple calls.

A practical example: a `Player` class that asks a `Dice` object to generate a roll.

```cpp
#include <iostream>

class Dice
{
public:
    int roll(int sides) const
    {
        return (sides > 0) ? (1 + (sides / 2)) : 0;
    }
};

class Player
{
    std::string m_name{};
public:
    Player(std::string name) : m_name{ name } {}
    void takeTurn(const Dice& d)
    {
        std::cout << m_name << " rolled " << d.roll(6) << '\n';
    }
};
```

`Player` does not own or store the `Dice`. It receives a `const Dice&` for the duration of `takeTurn` and never sees it again. That is a dependency -- the weakest coupling possible.

Why dependencies matter for design:

- **Loose coupling.** Because the dependent class only needs the other object temporarily, changes to the used class have minimal ripple effects.
- **Testability.** You can pass a different implementation (or a mock) into the function without touching the class's members.
- **Clarity of intent.** When you see a parameter instead of a member, you immediately know the relationship is transient.

How to distinguish the four relationships at a glance: if the object is created and destroyed inside the class, it is composition. If it is stored as a pointer/reference but created elsewhere, ask whether the class manages its lifetime (aggregation) or merely knows about it (association). If the object appears only as a parameter or local variable and is never stored, it is a dependency.

Prefer dependency over association whenever permanent access is unnecessary. Keeping relationships as weak as possible makes classes easier to understand, reuse, and test independently.
