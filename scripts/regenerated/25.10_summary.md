Virtual functions let you call derived behavior through a base pointer without knowing the concrete type. Occasionally, though, you need to go the other direction -- you have a `Base*` and need to obtain a `Derived*` so you can call a function that only exists in `Derived`. This is called **downcasting**, and C++ provides `dynamic_cast` to do it safely at runtime.

`dynamic_cast` works only on **polymorphic types** -- classes that have at least one virtual function (typically a virtual destructor). When you write `dynamic_cast<Derived*>(basePtr)`, the runtime checks whether the object actually is a `Derived`. If it is, you get a valid `Derived*`. If it is not, you get `nullptr`. This makes the failure mode explicit and easy to check.

```cpp
#include <iostream>
class Base {
public:
    virtual ~Base() = default;
};
class Derived : public Base {
public:
    std::string extra() const { return "Derived-only"; }
};
int main() {
    Derived d{};
    Base* bp{&d};
    Derived* dp{dynamic_cast<Derived*>(bp)};
    if (dp) std::cout << dp->extra() << '\n';
}
```

You can also `dynamic_cast` references. The syntax is `dynamic_cast<Derived&>(baseRef)`. Because a reference cannot be null, a failed reference cast throws `std::bad_cast` instead of returning nullptr. Wrap the attempt in a try-catch if the cast might legitimately fail.

Why not just use `static_cast`? A `static_cast` downcast compiles without complaint and performs no runtime check. If the object is not actually the target type, the resulting pointer is invalid and using it is undefined behavior. `dynamic_cast` trades a small runtime cost for safety.

Despite its usefulness, `dynamic_cast` is a tool of last resort. If you find yourself downcasting frequently, it usually signals that your base class interface is missing a virtual function. Prefer adding a virtual function to `Base` over scattering `dynamic_cast` calls throughout your code. Reserve `dynamic_cast` for cases where modifying the base class is impractical -- such as when you do not own it -- or when only one derived type needs special handling that does not belong in the common interface.
