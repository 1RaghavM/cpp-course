When you assign a Derived object to a Base variable **by value**, the compiler copies only the Base portion of the object. The derived data members and the derived virtual function table are discarded. The result is a plain Base object -- the derived parts have been "sliced" off. This is called **object slicing**.

```cpp
#include <iostream>
class Base {
public:
    virtual std::string_view identify() const { return "Base"; }
};
class Derived : public Base {
    int m_x{42};
public:
    std::string_view identify() const override { return "Derived"; }
    int getX() const { return m_x; }
};
int main() {
    Derived d{};
    Base b{d};           // slicing happens here
    std::cout << b.identify(); // prints "Base", not "Derived"
}
```

The assignment `Base b{d}` invokes the Base copy constructor, which only knows about Base members. The `m_x` member and the fact that the object was a Derived are both lost. Calling `identify()` on `b` returns `"Base"` because `b` is genuinely a Base object now, not a Derived pretending to be one.

Slicing is dangerous because it happens silently in three common situations. First, when you pass a Derived object **by value** to a function that takes a Base parameter. The function receives a sliced copy. Second, when you store Derived objects in a `std::vector<Base>` -- each element is copy-constructed as a Base, slicing every Derived that goes in. Third, when you accidentally assign a Derived to an existing Base variable.

The fix is straightforward: **use references or pointers** whenever you work with objects in an inheritance hierarchy. A `Base&` or `Base*` preserves the full derived object and virtual dispatch works correctly. For containers, use `std::vector<Base*>` or `std::vector<std::unique_ptr<Base>>` instead of `std::vector<Base>`.

Slicing does not cause a compiler error or warning in most cases. The program compiles and runs -- it just silently loses derived state and calls the wrong virtual functions. This makes it one of the more insidious bugs in C++ inheritance code. The rule of thumb is simple: never copy polymorphic objects by value.
