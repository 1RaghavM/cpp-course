# Comments

In the previous lesson, *Statements and the structure of a program*, you learned that a C++ program is built from statements inside a `main` function. As programs grow, the code alone doesn't always make the *purpose* clear. Comments let you leave notes for yourself and anyone else who reads your code later. The compiler ignores comments entirely — they produce no instructions and have zero effect on how the program runs.

## Single-line comments

A single-line comment starts with `//`. Everything from `//` to the end of that line is a comment.

```cpp
// This program does nothing useful yet.
// It just demonstrates comments.
int main()
{
    // The program ends here and reports success.
    return 0; // returning 0 means no error
}
```

You can place a `//` comment on its own line or at the end of a line that already contains a statement. Either way, the compiler skips everything after `//` on that line.

## Multi-line comments

When a note spans several lines, wrap it with `/*` to start and `*/` to end:

```
/* This is a multi-line comment.
   It can span as many lines as you need.
   The compiler ignores all of it. */
```

Multi-line comments cannot be nested. A `/*` inside an already-open `/* ... */` block does not start a second comment — the first `*/` the compiler encounters closes the block, which can cause unexpected errors.

## When to comment

Write a comment when the *reason* behind code is not obvious from the code itself. Good uses include:

- Explaining *why* a particular approach was chosen over an alternative.
- Summarizing what a section of code is responsible for at a high level.
- Noting anything surprising or non-obvious.

## What NOT to comment

Avoid restating what the code already says. A comment like `// return 0` next to `return 0;` adds nothing — anyone reading C++ can see the statement. Redundant comments create clutter and become a maintenance burden: when the code changes but the comment doesn't get updated, the comment becomes misleading, which is worse than having no comment at all.

A useful guideline: comments should explain *why*, not *what*. If the code is clear on its own, silence is better than noise.
