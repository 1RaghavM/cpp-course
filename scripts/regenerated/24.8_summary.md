Sometimes a derived class inherits member functions that should not be part of its public interface. C++ gives you two tools to restrict or reshape inherited functionality: **using declarations** to change access levels, and **= delete** to forbid calls entirely.

A `using` declaration inside a derived class's access section pulls a base class member into that section. If the base class has a public member and you place `using Base::member` inside the `private` or `protected` section of the derived class, that member becomes inaccessible (or restricted) through the derived type. You can also go the other direction -- if a base class member is `protected`, a `using` declaration in the `public` section of the derived class makes it publicly accessible.

```cpp
#include <iostream>

class Base {
public:
    int getValue() const { return m_val; }
protected:
    int m_val{};
};

class Restricted : public Base {
private:
    using Base::getValue; // now private in Restricted
public:
    void printDouble() const {
        std::cout << (getValue() * 2) << '\n';
    }
};
```

Here `Restricted` inherits `getValue()` publicly from `Base`, but the `using` declaration moves it into the `private` section. Code that holds a `Restricted` object cannot call `getValue()` directly, but the class itself can still use it internally. Note that if someone casts a `Restricted&` to a `Base&`, they can still call `getValue()` through the base reference -- hiding is not a security mechanism, it is an interface design tool.

The second technique is marking an inherited function as **deleted** in the derived class. Writing `void someFunction() = delete;` in the derived class causes any attempt to call that function on a derived object to produce a compile error. Unlike the access-level approach, a deleted function cannot even be called from inside the derived class.

Use these techniques when a derived class represents a narrower concept than the base. For example, a `FixedArray` derived from a resizable `Array` might delete the `resize()` function because fixed-size arrays must not change length. Hiding keeps the derived class's interface honest about what it actually supports.

Both techniques apply only when accessed through the derived type. A pointer or reference to the base type bypasses the hiding entirely. This is an inherent limitation of the non-virtual approach -- true interface enforcement across pointer types requires virtual functions (covered later).
