When you return a large object from a function, the compiler sees the return value as an r-value. In lesson 22.2 you learned that r-value references (`&&`) can bind to these temporaries. A **move constructor** and **move assignment operator** exploit this: instead of deep-copying a resource, they *steal* it from the source and leave the source in a safe empty state.

Consider a class that owns a dynamically allocated array:

```cpp
#include <iostream>

class IntArray {
    int* m_data{};
    int m_size{};
public:
    IntArray(int size) : m_data{ new int[size]{} }, m_size{ size } {}
    ~IntArray() { delete[] m_data; }
    IntArray(const IntArray& src) // copy ctor
        : m_data{ new int[src.m_size] }, m_size{ src.m_size } {
        for (int i{ 0 }; i < m_size; ++i) m_data[i] = src.m_data[i];
    }
    IntArray(IntArray&& src) noexcept // move ctor
        : m_data{ src.m_data }, m_size{ src.m_size } {
        src.m_data = nullptr; src.m_size = 0;
    }
};
```

The move constructor takes an `IntArray&&` parameter. It copies the pointer and size from `src`, then **nulls the source pointer** so the source's destructor does not free the stolen memory. Marking it `noexcept` is important because standard containers will only use move operations when they are guaranteed not to throw.

A **move assignment operator** follows the same idea but must also release any resource the target already owns:

```
IntArray& operator=(IntArray&& src) noexcept {
    if (this != &src) {
        delete[] m_data;          // free existing resource
        m_data = src.m_data;      // steal
        m_size = src.m_size;
        src.m_data = nullptr;     // null the source
        src.m_size = 0;
    }
    return *this;
}
```

The self-assignment check prevents an object from accidentally deleting its own data.

With the copy constructor, copy assignment, and destructor (the Rule of Three), plus the move constructor and move assignment operator, you now have the **Rule of Five**: if a class manages a resource, define or delete all five. The compiler calls the move versions automatically when the source is an r-value -- for example, a temporary returned from a function. No special cast is needed; the r-value nature of the expression selects the `&&` overload.

After a move, the source object must remain in a valid, destructible state. The convention is to null any owned pointers and zero out sizes, so the destructor runs harmlessly.
