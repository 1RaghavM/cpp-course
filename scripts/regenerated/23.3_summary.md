In the previous lesson, composition modeled a "has-a" relationship where the part cannot exist without the whole. **Aggregation** is the other flavor of "has-a": the whole *uses* its parts, but the parts have independent lifetimes. When the aggregate is destroyed, the parts continue to exist.

Consider a university department. A `Department` has teachers, but those teachers were hired before the department object was created and will continue to exist after it is dissolved. The department does not create or destroy its teachers -- it merely references them. This is aggregation.

The implementation difference from composition is straightforward: where composition uses member variables (or owned pointers that the class `new`s and `delete`s), aggregation uses **references or pointers to objects that live elsewhere**. The aggregate receives its parts through the constructor or a setter, but never allocates or frees them.

```cpp
#include <iostream>
#include <string>
class Teacher {
    std::string m_name;
public:
    Teacher(std::string name) : m_name{ name } {}
    const std::string& getName() const { return m_name; }
};
class Department {
    const Teacher& m_lead; // reference, not owned
public:
    Department(const Teacher& lead) : m_lead{ lead } {}
    void print() const { std::cout << "Lead: " << m_lead.getName(); }
};
```

Here `Department` stores a `const Teacher&`. It did not create the `Teacher`, and when the `Department` goes out of scope the `Teacher` is unaffected. The caller is responsible for ensuring the `Teacher` outlives the `Department`.

Key properties that distinguish aggregation from composition:

- The part can belong to **more than one aggregate** at the same time (a teacher can be in multiple departments).
- The aggregate does **not manage the lifetime** of its parts -- no `new`, no `delete`, no ownership transfer.
- The part has **no knowledge** of the aggregate. Removing the aggregate leaves the part intact.
- Implementation typically uses references (when the part is mandatory and never changes) or raw pointers (when the part is optional or reassignable).

Because the aggregate does not own its parts, you must be careful about dangling references. If the referenced object is destroyed before the aggregate, the aggregate holds an invalid reference. The simplest rule: ensure every referenced part outlives the aggregate that uses it. Designing clear ownership boundaries prevents this class of bug.
