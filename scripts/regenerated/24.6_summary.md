When you create a derived class, it inherits every member of its base. But inheritance is not just about reusing what already exists -- a derived class can **add entirely new member functions and member variables** that the base class knows nothing about. This is how you extend the interface beyond what the base provides.

Consider a base class `Fruit` that stores a name and a color. A derived class `Banana` can add a `bend()` member function and a `m_isPeeled` member variable that make no sense for a generic `Fruit`:

```cpp
#include <iostream>
#include <string>
#include <string_view>

class Fruit
{
private:
    std::string m_name{};
    std::string m_color{};
public:
    Fruit(std::string_view name, std::string_view color)
        : m_name{ name }, m_color{ color } {}
    std::string_view getName() const { return m_name; }
    std::string_view getColor() const { return m_color; }
};
```

The derived class adds its own state and behavior:

```cpp
class Banana : public Fruit
{
private:
    bool m_isPeeled{ false };
public:
    Banana(std::string_view color)
        : Fruit{ "Banana", color } {}
    void peel() { m_isPeeled = true; }
    bool isPeeled() const { return m_isPeeled; }
};
```

`Banana` objects can call `getName()` and `getColor()` (inherited from `Fruit`) **and** `peel()` and `isPeeled()` (defined only in `Banana`). A plain `Fruit` object cannot call `peel()` because that function does not exist in `Fruit`'s interface.

This is a key insight: derived classes are not limited to what the base defines. They can introduce any members they need. The base class does not need to be modified, and adding members to a derived class never affects the base or other derived classes.

Note that new member variables added in the derived class should almost always be `private`. They are initialized in the derived class's member initializer list (or via a default member initializer), since the base class constructor has no knowledge of them.

New member functions in the derived class can access inherited `public` and `protected` members of the base, but not `private` base members -- the same access rules you learned in the prior lessons on access specifiers still apply.
