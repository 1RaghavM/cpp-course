## Destructors and Dynamic Memory

In Ch 15.4 you learned that a **destructor** is a special member function that runs automatically when an object is destroyed. With dynamic allocation in the picture, destructors become essential: they are where you `delete` memory that the class allocated with `new`, preventing leaks.

This pattern has a name: **RAII** (Resource Acquisition Is Initialization). The constructor acquires a resource (allocates memory), and the destructor releases it (deallocates memory). Because destructors run automatically when an object leaves scope, cleanup is guaranteed even if an early `return` or exception exits the function.

```cpp
#include <iostream>

class IntArray
{
    int* m_data{};
    int m_size{};
public:
    IntArray(int size)
        : m_data{ new int[size]{} }, m_size{ size } {}

    ~IntArray()
    {
        delete[] m_data;
        std::cout << "Freed " << m_size << " ints\n";
    }

    int size() const { return m_size; }
};
```

When an `IntArray` object goes out of scope, `~IntArray()` fires and calls `delete[]` on `m_data`. The caller never needs to remember to free anything.

### Order of destruction

Local objects are destroyed in the **reverse order** of their construction. If you create `a` then `b`, the destructor for `b` runs first, then `a`. This reverse ordering ensures that if `b` depends on `a`, `b` is cleaned up while `a` is still valid.

For class members, destruction also happens in reverse declaration order, after the body of the destructor finishes executing.

### Guidelines

- If your class calls `new` in the constructor, it must call `delete` in the destructor (and `new[]` pairs with `delete[]`).
- A destructor should never throw exceptions. If deallocation cannot fail, this is naturally satisfied.
- Setting the pointer to `nullptr` after `delete` inside a destructor is unnecessary since the object is being destroyed anyway.
- If you find yourself writing manual `delete` calls outside a destructor, consider whether RAII can handle it instead.
