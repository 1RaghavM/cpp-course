In lesson 13.5 you overloaded `operator<<` for enum types. The same technique works for classes, but with a twist: classes typically have private members, so the overload needs special access.

`operator<<` and `operator>>` cannot be member functions of your class. The left operand of `std::cout << obj` is an `std::ostream`, not your class, so the function must be a non-member. To let it read private data, declare it as a **friend** inside the class.

Both overloads must return a reference to the stream so that calls can be chained (`std::cout << a << b`). The output overload takes `std::ostream&` and a `const` reference to your object. The input overload takes `std::istream&` and a non-const reference, because it modifies the object.

```cpp
#include <iostream>

class Point {
    double m_x{};
    double m_y{};
public:
    Point(double x, double y) : m_x{x}, m_y{y} {}
    friend std::ostream& operator<<(std::ostream& out, const Point& p);
    friend std::istream& operator>>(std::istream& in, Point& p);
};

std::ostream& operator<<(std::ostream& out, const Point& p) {
    out << '(' << p.m_x << ", " << p.m_y << ')';
    return out;
}

std::istream& operator>>(std::istream& in, Point& p) {
    in >> p.m_x >> p.m_y;
    return in;
}
```

With these overloads defined, `Point` objects work seamlessly with `std::cout`, `std::cin`, file streams, and string streams -- any class derived from `std::ostream` or `std::istream`.

The pattern is always the same: declare the overload as a friend, accept the stream by reference, perform the I/O, and return the stream. Keep the output format simple and consistent so that what `operator>>` reads matches what `operator<<` writes whenever round-tripping matters.

One common mistake is forgetting to return the stream reference. Without the return, chained expressions like `std::cout << a << b` will not compile. Another mistake is taking the object by value instead of by reference, which causes unnecessary copies and prevents `operator>>` from modifying the original.
