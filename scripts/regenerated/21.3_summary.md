In lesson 21.2 you learned to overload operators as friend functions, which grants direct access to private members. But friendship is not always necessary -- if a class already exposes public getters for the data an operator needs, you can overload that operator as an ordinary non-member, non-friend function. This keeps the class interface tighter because fewer functions can touch its internals.

A normal-function overload looks identical to a friend overload except that the declaration does not appear inside the class and is not prefixed with `friend`. Because it has no special access, the implementation must use the class's public interface.

```cpp
#include <iostream>

class Cents {
    int m_cents{};
public:
    Cents(int cents) : m_cents{ cents } {}
    int getCents() const { return m_cents; }
};

Cents operator+(const Cents& a, const Cents& b) {
    return Cents{ a.getCents() + b.getCents() };
}

int main() {
    Cents sum{ Cents{6} + Cents{4} };
    std::cout << sum.getCents() << '\n'; // 10
}
```

The `operator+` above is a free function in the global namespace. It accesses each operand through `getCents()`, so no friendship is required. The compiler finds it through argument-dependent lookup (ADL) because at least one parameter is of type `Cents`.

So when should you prefer a normal function over a friend? Use the normal form when the class already provides sufficient public access. Use `friend` only when the operator genuinely needs to reach private or protected members that no getter exposes. This follows the general principle of granting the least privilege necessary -- the fewer functions that can see private state, the easier the class is to maintain and reason about.

One practical consequence: if you later change the class's internal representation, only the member functions and friends need updating. Normal-function overloads, which depend only on the public API, remain untouched. This makes the normal-function approach more resilient to refactoring.

Note that not every operator can be overloaded as a non-member. The assignment (`=`), subscript (`[]`), function-call (`()`), and member-access (`->`) operators must be member functions. For the symmetric binary operators like `+`, `-`, `*`, and `==`, non-member overloads are usually the better default because they allow implicit conversion on both operands equally.
