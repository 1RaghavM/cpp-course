## std::unique_ptr

`std::unique_ptr` is a smart pointer that enforces **exclusive ownership**: exactly one `unique_ptr` owns the managed object at any time. When the `unique_ptr` is destroyed (goes out of scope), it automatically `delete`s the object it holds. This is the RAII pattern applied to dynamic memory -- acquisition happens in the constructor, release happens in the destructor, and you never write `delete` yourself.

Create a `unique_ptr` with `std::make_unique<T>(args...)`, which allocates the object and wraps it in one expression. Prefer this over raw `new` because it is exception-safe and avoids writing the type twice.

Because ownership is exclusive, `unique_ptr` **cannot be copied** -- its copy constructor and copy assignment operator are deleted. Attempting `auto b = a;` where `a` is a `unique_ptr` produces a compile error. To transfer ownership, use `std::move`, which converts the source to an rvalue so the move constructor takes over. After the move, the source becomes `nullptr`.

You access the managed object through `operator->` (member access), `operator*` (dereference), or `.get()` (raw pointer without releasing ownership). To check whether a `unique_ptr` currently owns an object, compare it to `nullptr` or use it in a boolean context.

```cpp
#include <iostream>
#include <memory>

struct Sensor {
    int id;
    Sensor(int id) : id{id} { std::cout << "Sensor " << id << " online\n"; }
    ~Sensor() { std::cout << "Sensor " << id << " offline\n"; }
};

int main() {
    auto s = std::make_unique<Sensor>(5);
    std::cout << "Reading sensor " << s->id << '\n';
    auto t = std::move(s);        // ownership transferred
    std::cout << std::boolalpha << (s == nullptr) << '\n'; // true
}   // Sensor 5 offline -- automatic cleanup
```

Key guidelines: default to `unique_ptr` for any heap allocation. Pass raw pointers or references to functions that observe but do not own the object -- call `.get()` or dereference at the call site. Reserve `std::shared_ptr` for the rare case where multiple owners genuinely share lifetime responsibility. A `unique_ptr` has zero overhead compared to a raw pointer -- no reference count, no extra allocation -- so there is no reason to avoid it.
