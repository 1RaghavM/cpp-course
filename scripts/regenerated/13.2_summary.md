## Unscoped Enumerations

An **unscoped enumeration** (defined with the `enum` keyword) creates a distinct type whose values are restricted to a set of named constants called **enumerators**. Each enumerator is an integral constant, and by default the compiler assigns values starting at `0` and incrementing by one.

```cpp
#include <iostream>

enum Color
{
    red,    // 0
    green,  // 1
    blue    // 2
};

int main()
{
    Color shirt{ green };
    std::cout << shirt << '\n'; // prints 1
}
```

The name `Color` is the **enumerated type**, and `red`, `green`, `blue` are its enumerators. Because `Color` is a distinct type, you cannot assign a raw integer to it directly: `Color c{ 5 };` is a compile error.

### Explicit Values

You can assign specific integer values to any enumerator with `= value`. Enumerators without an explicit value receive the prior enumerator's value plus one. For example, if you write `fileNotFound = 10`, the next enumerator without an explicit value would be `11`. Duplicate values are legal but usually a sign of poor design.

### Implicit Conversion to int

Unscoped enumerators **implicitly convert** to their underlying integer type. That is why `std::cout << shirt` prints `1` rather than `green`. This implicit conversion is convenient but can mask bugs -- you can accidentally pass an enumerator where a plain `int` is expected and the compiler will not warn you.

### Naming Conventions

- The enum type name uses PascalCase (`Color`, `Direction`).
- Enumerator names typically use camelCase or snake_case starting with a lowercase letter. Some codebases prefix enumerators with the type name (e.g., `color_red`) to avoid name collisions, since unscoped enumerators are placed into the same scope as the enum itself.

### Scope Behavior

The word "unscoped" matters: the enumerators are injected into the **surrounding scope**, not confined inside the enum type. If two unscoped enums in the same scope both define an enumerator named `red`, you get a naming collision. This is the primary drawback of unscoped enums and the motivation for scoped enums (`enum class`), covered in a later lesson.

### When to Use Unscoped Enums

Use them when you have a small, fixed set of related named constants and you want the convenience of implicit integer conversion -- for example, representing cardinal directions, simple status codes, or menu options.
