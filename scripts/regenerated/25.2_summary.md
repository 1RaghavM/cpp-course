In lesson 25.1 you saw the problem: when a function takes a `Base&` or `Base*` and you pass a `Derived` object, calls to member functions resolve to the **Base version** at compile time. The derived override is ignored. This is called **early binding** (or static dispatch) and it is the default in C++.

The `virtual` keyword fixes this. When you mark a Base member function as `virtual`, the compiler switches from early binding to **late binding** (dynamic dispatch). Now the version of the function that gets called is determined at runtime based on the **actual type of the object**, not the type of the pointer or reference.

```cpp
#include <iostream>
#include <string_view>
class Animal {
public:
    virtual std::string_view speak() const { return "..."; }
};
class Cat : public Animal {
public:
    std::string_view speak() const { return "Meow"; }
};
class Dog : public Animal {
public:
    std::string_view speak() const { return "Woof"; }
};
void report(const Animal& a) { std::cout << a.speak() << '\n'; }
```

Calling `report(cat)` now prints `Meow` and `report(dog)` prints `Woof`, even though `report` only knows about `Animal&`. Remove the `virtual` keyword and both calls print `...` -- the exact problem from 25.1.

Once a function is declared `virtual` in the base class, it is automatically virtual in every derived class whether or not you repeat the keyword. Good practice is to always repeat it for clarity, but it is not required.

This unlocks **polymorphism** -- the ability to write code that operates on a collection of different types through a common base interface. You can store an array of `Animal*` where each element points to a different derived type, loop over them, and call `speak()` on each one. The correct derived version executes every time. This is the core pattern behind virtually every plugin system, game entity loop, and UI widget hierarchy in C++.

Two rules to remember: virtual dispatch only works through **pointers or references** to the base class. If you copy a Derived into a Base variable by value, the derived parts are lost (a topic for a later lesson). Second, the function signatures in Base and Derived must match exactly -- same name, same parameters, same const qualification -- or the derived version will not override; it will hide the base version instead.
