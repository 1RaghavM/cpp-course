When a derived class object is created, the base class portion must be constructed first. Since the base class may need arguments to initialize its members, the derived class constructor must **pass those arguments to the base class constructor** through its member initializer list.

A derived class **cannot directly initialize inherited members**. Those members belong to the base class and must be initialized by the base class constructor. Attempting to set them in the derived class initializer list is a compiler error. Instead, you delegate to the base constructor:

```cpp
class Base {
    int m_id{};
public:
    Base(int id) : m_id{id} {}
    int getId() const { return m_id; }
};

class Derived : public Base {
    double m_value{};
public:
    Derived(int id, double value)
        : Base{id}, m_value{value} {}
};
```

The syntax `Derived(int id, double value) : Base{id}, m_value{value}` first calls `Base(int)` to initialize the inherited portion, then initializes the derived member `m_value`. The base constructor always runs before any derived member initializers, regardless of the order you write them.

If you omit the base class from the member initializer list, the compiler attempts to call the base class **default constructor**. If the base has no default constructor, this is a compile error.

This pattern extends to **chains of inheritance**. If class `C` derives from `B`, which derives from `A`, then constructing a `C` object calls `A`'s constructor first, then `B`'s, then `C`'s. Each class in the chain is responsible only for passing arguments to its immediate parent:

`C(int x, int y, int z) : B{x, y}, m_z{z} {}` and inside `B`: `B(int x, int y) : A{x}, m_y{y} {}`.

The key takeaway is that each class in a hierarchy initializes only its own members and delegates upward to its base class constructor for everything it inherited. This keeps each class responsible for exactly the data it introduces.
