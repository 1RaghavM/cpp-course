## Passing and Returning std::array

Because `std::array` is a class type, passing it by value copies every element. For any non-trivial size this is wasteful, so the standard practice is to pass by **const reference** when the function only reads the data.

```cpp
#include <array>
#include <cstddef>
#include <iostream>

template <typename T, std::size_t N>
void printArray(const std::array<T, N>& arr)
{
    for (std::size_t i{ 0 }; i < N; ++i)
        std::cout << arr[i] << ' ';
    std::cout << '\n';
}

int main()
{
    std::array a{ 10, 20, 30, 40 };
    printArray(a);
}
```

### Why a function template?

The element type **and** the length are both part of `std::array`'s type. A function that accepts `const std::array<int, 4>&` refuses a `std::array<int, 5>`. To write a function that works with any size, make it a template parameterized on `T` and `N` (a non-type template parameter of type `std::size_t`). Inside the function body you can use `N` directly instead of calling `.size()`.

If the function must modify elements, drop the `const` and pass by plain reference (`std::array<T, N>&`).

### Returning std::array

Returning a `std::array` by value is safe and idiomatic. The compiler applies copy elision (or move semantics) so no redundant copies are made in practice. Simply declare the return type and return a local or brace-initialized array.

### Why std::array beats C-style arrays here

C-style arrays silently decay to pointers when passed to a function, losing their length information entirely. `std::array` never decays: the function always knows both the element type and the size at compile time. This is one of the strongest reasons to prefer `std::array` over C-style arrays in modern C++.

### Guidelines

- Pass by `const&` for read-only access, by `&` when the function must modify elements.
- Use a function template when the function should work with multiple array sizes.
- Return `std::array` by value freely -- the compiler optimizes away the copy.
