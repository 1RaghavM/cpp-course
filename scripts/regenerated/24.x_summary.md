## Chapter 24 Summary

This chapter covered inheritance, the mechanism that lets one class reuse and extend another class's members without duplicating code.

### Introduction and basic mechanics (24.1, 24.2)

Inheritance models an "is-a" relationship: a `Dog` is an `Animal`, a `Car` is a `Vehicle`. You write `class Derived : public Base` to make `Derived` inherit all non-private members of `Base`. The derived class gets the base class's public and protected member variables and member functions automatically. What is **not** inherited: constructors, destructors, and assignment operators.

### Construction and destruction order (24.3)

When a derived object is created, the base portion is constructed first, then the derived portion. Destruction happens in reverse: derived destructor runs first, then the base destructor. This guarantees that inherited members are fully initialized before the derived constructor body executes, and still intact when the derived destructor runs.

### Constructors and initialization (24.4)

Because constructors are not inherited, the derived class must explicitly call a base constructor in its member initializer list. If you omit this call, the compiler attempts to use the base class's default constructor. If no default constructor exists, compilation fails.

### Access specifiers and inheritance (24.5)

The type of inheritance (public, protected, private) controls how base members are exposed through the derived class. With public inheritance, public base members stay public and protected members stay protected. With protected inheritance, both become protected in the derived class. With private inheritance, both become private. In practice, public inheritance is used almost exclusively.

### Adding and overriding functionality (24.6, 24.7)

Derived classes can add entirely new member functions and member variables. They can also override inherited functions by defining a function with the same name and signature. Inside the overriding function, you can call the base version explicitly using `Base::functionName()` to extend rather than completely replace the inherited behavior.

### Hiding inherited functionality (24.8)

You can hide a base class function in the derived class by marking it `= delete` or by changing its access level using a `using` declaration. A `using Base::functionName` declaration in a different access section of the derived class changes the visibility of that function. Marking an inherited function `= delete` prevents it from being called through the derived class entirely.

### Multiple inheritance (24.9)

C++ allows a class to inherit from more than one base class: `class Derived : public Base1, public Base2`. This introduces the diamond problem: if two base classes share a common ancestor, the derived class ends up with two copies of the ancestor's members, creating ambiguity. While C++ provides solutions (such as explicit scope resolution), multiple inheritance adds significant complexity and should be used sparingly.
