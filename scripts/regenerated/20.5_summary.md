C inherited a mechanism called **ellipsis** (`...`) that lets a function accept a variable number of arguments. You have already used one such function without realizing it: `std::printf`. The signature looks like `int printf(const char* fmt, ...)` -- the ellipsis means "zero or more additional arguments of any type."

To write your own variadic function you include `<cstdarg>` and use four macros: `va_list` declares an argument list object, `va_start` initializes it, `va_arg` retrieves the next argument (you must supply its type), and `va_end` cleans up.

```cpp
#include <cstdarg>
#include <iostream>

double sumDoubles(int count, ...)
{
    va_list args;
    va_start(args, count);
    double total{ 0.0 };
    for (int i{ 0 }; i < count; ++i)
        total += va_arg(args, double);
    va_end(args);
    return total;
}
```

This compiles and works when used correctly, but the compiler cannot verify that the caller actually passes `double` values or the right number of them. Passing an `int` where `double` is expected causes **undefined behavior** -- no warning, no error, just silent data corruption. There is also no way to query how many arguments were actually passed; the function must rely on the caller to communicate this correctly (here via the `count` parameter).

These are fundamental, unfixable problems with ellipsis:

- **No type checking:** the compiler skips type validation for every argument matched by `...`.
- **No count checking:** passing too few or too many arguments is not caught at compile time.
- **Class types are dangerous:** passing most non-trivial class objects (like `std::string`) through ellipsis is undefined behavior.

Modern C++ provides strictly superior alternatives that preserve type safety. Function overloading (which you already know) handles cases with a small, fixed set of variants. For genuinely variable argument counts, `std::initializer_list` and templates (covered in later chapters) solve the problem with full compile-time type checking. There is virtually no reason to write a new C-style variadic function in modern C++ -- prefer these alternatives in every case.
