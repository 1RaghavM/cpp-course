In lesson 21.1 you learned that operator overloading lets you define how operators work with your own types. The arithmetic operators (`+`, `-`, `*`, `/`) are among the most commonly overloaded. The cleanest way to implement them is as **friend functions** -- non-member functions that have direct access to a class's private members.

A friend arithmetic operator takes two parameters (the left and right operands), performs the computation, and **returns the result by value**. Returning by value is essential because the result is a new object -- it does not modify either operand. Both parameters should be `const` references to avoid unnecessary copies and to allow the operator to work with const objects.

```cpp
#include <iostream>

class Cents {
private:
    int m_cents{};
public:
    Cents(int cents) : m_cents{cents} {}
    friend Cents operator+(const Cents& a, const Cents& b);
    int getCents() const { return m_cents; }
};

Cents operator+(const Cents& a, const Cents& b) {
    return Cents{a.m_cents + b.m_cents};
}

int main() {
    Cents sum{Cents{12} + Cents{8}};
    std::cout << sum.getCents() << '\n'; // 20
}
```

The `friend` declaration inside the class grants the standalone `operator+` function access to `m_cents`. The function itself is defined outside the class and is not a member -- it lives at namespace scope.

The pattern is identical for `-`, `*`, and `/`: replace the `+` with the desired operator and change the arithmetic inside the body. Each overload is independent, so you only define the ones that make sense for your type.

Why friend functions instead of member functions? When a binary operator is a member, the left operand must be an object of that class. A friend function treats both sides symmetrically, which matters once you start mixing types (e.g., `Cents + int`). Friend functions also keep the operator logic out of the class definition, reducing clutter.

Key rules to remember: always pass operands by `const&`, always return the new object by value, and only grant friendship for operators that genuinely need access to private data. If the operator can be implemented entirely through the public interface, prefer a plain non-member non-friend function instead.
