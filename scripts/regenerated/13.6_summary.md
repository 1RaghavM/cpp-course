## Scoped Enumerations (enum classes)

Unscoped enumerations (`enum`) place their enumerators in the surrounding scope and implicitly convert to integers. This causes two recurring problems: name collisions between enums that share an enumerator name, and accidental arithmetic on values that should not behave like numbers. **Scoped enumerations** (`enum class`) fix both.

### Declaration and Access

Declare with `enum class` instead of `enum`. Enumerators are accessed through the enum name using the scope resolution operator:

```cpp
#include <iostream>
#include <string>

enum class Color { red, green, blue };
enum class Fruit { red, green, blue };  // no collision with Color::red

std::string colorToString(Color c) {
    switch (c) {
        case Color::red:   return "red";
        case Color::green: return "green";
        case Color::blue:  return "blue";
        default:           return "unknown";
    }
}
```

Both `Color` and `Fruit` define `red`, `green`, `blue` without conflict because each enumerator lives inside its own scope.

### No Implicit Conversion to int

Unlike unscoped enums, a scoped enumerator does **not** implicitly convert to an integer. Passing `Color::red` where an `int` is expected is a compile error. When you genuinely need the underlying integer, use `static_cast`:

```cpp
int index = static_cast<int>(Color::green); // 1
```

This forces every int conversion to be explicit, making accidental misuse visible in code review.

### Specifying the Underlying Type

By default the underlying type is `int`. You can choose a different integral type after a colon:

```cpp
enum class Permission : std::uint8_t { read = 1, write = 2, execute = 4 };
```

This is useful when memory matters or when you need a specific width for serialization.

### When to Use enum class vs. enum

Prefer `enum class` in new code. The scoping and type safety it provides eliminate entire categories of bugs. Use unscoped `enum` only when you have a deliberate need for implicit integer conversion (rare in modern C++).

### Key Takeaways

- `enum class` enumerators require scope resolution (`Color::red`, not just `red`)
- No implicit conversion to `int` — use `static_cast` when needed
- Two `enum class` types can share enumerator names without conflict
- You can specify the underlying type with `: <type>` after the enum name
