When you delete a derived object through a base-class pointer, the compiler must decide which destructor to call. If the base destructor is **not** virtual, only the base destructor runs. The derived destructor is skipped entirely, so any resources the derived class acquired -- memory, file handles, connections -- are leaked. This is undefined behavior by the standard and one of the most common polymorphism bugs in C++.

The fix is simple: whenever a class is designed to be inherited from and used through base pointers, **make its destructor virtual**.

```cpp
#include <iostream>
class Base {
public:
    virtual ~Base() { std::cout << "~Base\n"; }
};
class Derived : public Base {
public:
    ~Derived() override { std::cout << "~Derived\n"; }
};
int main() {
    Base* ptr = new Derived{};
    delete ptr; // prints ~Derived then ~Base
}
```

With the virtual destructor, `delete ptr` calls `~Derived()` first, then `~Base()` -- the normal destruction order. Without `virtual`, only `~Base()` would run.

A good rule of thumb: if a class has **any** virtual function, it should also have a virtual destructor. The compiler-generated default destructor is non-virtual, so you must declare it explicitly (even if the body is empty or defaulted with `= default`).

You can mark a destructor `override` in derived classes to state intent, and you can mark it `final` to prevent further derived classes from overriding the destructor. A class with a `final` destructor can still be inherited from, but the inheriting class cannot provide its own destructor body -- it must rely on the compiler-generated one. In practice, `final` on destructors is rare; marking the entire class `final` is more common when you want to block further derivation.

**Virtual assignment** follows a similar principle. If you declare `operator=` as `virtual` in the base class, derived classes can override it. However, virtual assignment is tricky because the parameter types differ between base and derived versions, making true polymorphic assignment error-prone. Most C++ code avoids virtual `operator=` and instead uses non-virtual assignment with virtual helper functions (like a virtual `clone()` pattern). For now, be aware that assignment operators **can** be virtual but rarely should be.

Key takeaway: always give your polymorphic base classes a virtual destructor. It is the single most important rule for safe polymorphic resource management.
