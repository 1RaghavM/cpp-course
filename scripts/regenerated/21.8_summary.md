C++ lets you overload the increment (`++`) and decrement (`--`) operators for your own types. The tricky part is that each operator has two forms -- prefix (`++x`) and postfix (`x++`) -- and the language uses a **dummy `int` parameter** to tell the compiler which overload is which.

### Prefix: return a reference

Prefix `++` modifies the object and returns the updated object by reference. This matches built-in behavior where `++x` is an lvalue you can continue to use in the same expression.

```cpp
#include <iostream>

class Counter
{
    int m_value{};
public:
    explicit Counter(int v) : m_value{ v } {}
    Counter& operator++()   { ++m_value; return *this; }
    Counter& operator--()   { --m_value; return *this; }
    int value() const       { return m_value; }
};

int main()
{
    Counter c{ 5 };
    std::cout << (++c).value() << '\n'; // 6
    std::cout << (--c).value() << '\n'; // 5
}
```

The function modifies the member, then returns `*this`. Because it returns by reference, no copy is made.

### Postfix: dummy int, return by value

Postfix `x++` must return the **old** value while still incrementing the object. To distinguish it from the prefix version, the postfix overload declares a dummy `int` parameter that the compiler supplies automatically (you never pass it yourself). Because the old value is a local snapshot, postfix returns by value, not by reference.

The canonical implementation stores a copy of the current object, calls the prefix version to do the actual increment, then returns the saved copy. This avoids duplicating the mutation logic.

```
Counter Counter::operator++(int) { Counter old{ *this }; ++(*this); return old; }
```

The same pattern applies to postfix `--`.

### Why prefix is preferred

Postfix creates a temporary copy, prefix does not. For lightweight types the difference is negligible, but as a habit prefer `++x` over `x++` when you do not need the old value. This convention carries over to iterators and other heavier types where the copy cost is real.
