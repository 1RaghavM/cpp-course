In the previous lesson we surveyed the types of object relationships in C++. Now we focus on the most common and most intuitive one: **composition**. Composition models a "part-of" relationship where the whole *owns* its parts. The parts have no meaningful existence outside the whole, and the whole is responsible for creating and destroying them.

Think of a body and a heart. The heart is part of the body. It is created when the body is created, and it is destroyed when the body is destroyed. The heart cannot belong to two bodies at the same time.

In C++ code, composition is implemented using **member variables with value semantics**. The part is a direct member (not a pointer or reference), so its lifetime is automatically tied to the containing object:

```cpp
struct Point2D
{
    int m_x{};
    int m_y{};
};
class Creature
{
    std::string m_name;   // composed part
    Point2D m_location;   // composed part
public:
    Creature(std::string_view name, int x, int y)
        : m_name{name}, m_location{x, y} {}
    const Point2D& getLocation() const { return m_location; }
};
```

Here `Creature` *composes* both a `std::string` and a `Point2D`. Neither member is allocated with `new`; both live inside the `Creature` object itself. When a `Creature` is copied, its parts are copied. When it is destroyed, its parts are destroyed automatically in reverse order of declaration.

The defining properties of composition are:

- **The part is a member of exactly one whole** at any time.
- **The whole manages the part's lifetime** -- construction, copying, and destruction happen through the whole.
- **The part has no independent identity** -- it cannot be accessed except through the whole (unless the whole explicitly exposes it).

Because C++ member variables already follow these rules by default, composition is the relationship you have been writing all along whenever you gave a class a data member. The key insight of this lesson is recognizing that pattern by name and understanding why it matters: it produces the tightest coupling between objects and the strongest lifetime guarantee, which makes code easier to reason about and less prone to dangling-reference bugs.
