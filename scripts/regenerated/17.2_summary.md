## std::array Length and Indexing

A `std::array` knows its own length. The `.size()` member function returns the number of elements as a `std::size_t` (an unsigned integer type). Because the length of a `std::array` is part of its type, `.size()` is a `constexpr` function -- the compiler can evaluate it at compile time.

```cpp
#include <array>
#include <iostream>

int main()
{
    constexpr std::array nums{ 10, 20, 30, 40, 50 };
    constexpr std::size_t len{ nums.size() }; // 5, compile-time

    std::cout << "length: " << len << '\n';
    std::cout << "first:  " << nums[0] << '\n';
    std::cout << "last:   " << nums.at(len - 1) << '\n';
}
```

### operator[] vs .at()

`std::array` provides two ways to access elements by index. `operator[]` performs **no bounds checking** -- if you pass an out-of-range index, the behavior is undefined, just like raw arrays. The `.at()` member function performs **runtime bounds checking** and throws a `std::out_of_range` exception if the index is invalid.

Use `operator[]` in performance-sensitive code where you have already validated the index yourself. Use `.at()` when correctness is more important than speed, or during debugging when you want to catch off-by-one errors immediately rather than silently reading garbage.

### constexpr size

Because `.size()` is `constexpr`, you can use it anywhere a compile-time constant is required -- for example, as a template argument or in a `static_assert`. This is a major advantage over C-style arrays, where determining the length requires a manual calculation. You can also store the result in a `constexpr` variable for clarity.

### Common pitfall: signed/unsigned mismatch

`.size()` returns `std::size_t`, which is unsigned. Comparing it directly with a signed integer (like `int`) triggers a compiler warning. To avoid this, either use `std::size_t` for your loop variable or cast explicitly. This will become even more relevant when we cover loops over arrays in a later lesson.

### Key takeaways

- `.size()` gives the element count and is usable at compile time.
- `operator[]` is fast but unchecked; `.at()` is safe but slightly slower.
- Prefer `.at()` while learning; switch to `operator[]` once your indexing logic is proven correct.
