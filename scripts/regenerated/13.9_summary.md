## Default Member Initialization

When you define a struct, each member can be given a **default member initializer** — a value written directly in the struct definition that applies whenever no explicit initializer is provided for that member.

```cpp
#include <iostream>
#include <string>

struct Config {
    std::string name{ "untitled" };
    int width{ 800 };
    int height{ 600 };
    bool fullscreen{ false };
};

int main() {
    Config c1{};              // all defaults: "untitled", 800, 600, false
    Config c2{ "game", 1920 }; // name="game", width=1920, height=600, fullscreen=false
    std::cout << c1.name << " " << c1.width << "x" << c1.height << '\n';
    std::cout << c2.name << " " << c2.width << "x" << c2.height << '\n';
}
```

Without default member initializers, members of fundamental types (like `int` or `bool`) are left uninitialized when you create an object without explicitly setting every field — reading them is undefined behavior. Default member initializers eliminate that risk at the point of definition.

### How Defaults Interact with Aggregate Initialization

When you aggregate-initialize a struct, any value you explicitly provide **overrides** the default for that member. Members you omit use their default. If a member has no default member initializer and you omit it during aggregate initialization, it is value-initialized (zero for fundamental types). This means `Config c2{ "game", 1920 };` sets `name` and `width` from the braces, while `height` and `fullscreen` keep their defaults of `600` and `false`.

Designated initializers work the same way — `Config c3{ .fullscreen = true }` sets only `fullscreen` to `true` while `name`, `width`, and `height` retain their defaults.

### Best Practice

Provide a default member initializer for every member. This guarantees that no matter how an object is created — value-initialized with `{}`, partially initialized, or even default-initialized — every member holds a meaningful value. The explicit initializer in the struct definition serves as living documentation of the "sensible default" for each field, making the type safer and easier to use correctly.
