In previous lessons every virtual function had a body -- a default implementation the base class provided. But sometimes a base class exists only to define an interface, and there is no sensible default. A `Shape` base class cannot compute a meaningful `area()` because "shape" is not a concrete thing. C++ handles this with **pure virtual functions**.

You declare a pure virtual function by adding `= 0` to the declaration:

```cpp
#include <iostream>
class Shape {
public:
    virtual double area() const = 0;   // pure virtual
    virtual ~Shape() = default;
};
class Circle : public Shape {
    double m_r{};
public:
    Circle(double r) : m_r{r} {}
    double area() const override { return 3.14159 * m_r * m_r; }
};
class Rectangle : public Shape {
    double m_w{}, m_h{};
public:
    Rectangle(double w, double h) : m_w{w}, m_h{h} {}
    double area() const override { return m_w * m_h; }
};
```

A class that contains at least one pure virtual function is an **abstract base class** (ABC). You cannot create an instance of an abstract class -- `Shape s{};` is a compiler error. The only purpose of an ABC is to serve as a base that derived classes inherit from. Every concrete derived class **must** override all inherited pure virtual functions; if it fails to override even one, it too becomes abstract and cannot be instantiated.

This gives you a powerful design tool: the base class defines a **contract** (which functions must exist and what signatures they must have), and the compiler enforces that every concrete derived class fulfills that contract. Code that accepts a `Shape&` or `Shape*` can call `area()` with confidence that a real implementation exists.

An **interface class** is an abstract class that has no member variables, no non-pure-virtual functions (except the destructor), and a virtual destructor. It defines pure behavior with zero state. Interface classes are the C++ equivalent of interfaces in languages like Java. A typical interface class contains only pure virtual functions and a virtual destructor defaulted or with an empty body.

Best practices: always give your abstract base class a **virtual destructor** so that deleting through a base pointer correctly destroys derived objects. Use `override` on every derived implementation. Prefer an interface class when you only need to specify behavior, and use an ABC with some shared implementation when derived classes share common code.
