When a class inherits from a base class, the **access specifier** on the inheritance declaration controls how base members are exposed through the derived class. C++ supports three inheritance specifiers -- `public`, `protected`, and `private` -- and a corresponding `protected` access level for member declarations.

**The protected access level.** A `protected` member behaves like `private` to outside code: external callers cannot access it. But unlike `private`, a `protected` member *is* directly accessible in any derived class. This lets base classes share implementation details with children without exposing them publicly.

**How inheritance access specifiers work.** The specifier sets an upper bound on how base members appear in the derived class:

- **Public inheritance** (`class Derived : public Base`): base `public` stays `public`, base `protected` stays `protected`, base `private` remains inaccessible. The derived class fully substitutes for the base. Use this almost always.
- **Protected inheritance** (`class Derived : protected Base`): base `public` becomes `protected`, base `protected` stays `protected`, base `private` remains inaccessible.
- **Private inheritance** (`class Derived : private Base`): base `public` and `protected` both become `private`, base `private` remains inaccessible. Outside code sees none of the inherited interface.

Regardless of inheritance type, the derived class **never** gains access to the base's `private` members. Access specifiers restrict visibility; they never expand it.

```cpp
#include <iostream>

class Base
{
protected:
    int m_value{};
public:
    Base(int v) : m_value{ v } {}
    int getValue() const { return m_value; }
};

class Derived : public Base
{
public:
    Derived(int v) : Base{ v } {}
    bool isPositive() const { return m_value > 0; }
};
```

`Derived` reads `m_value` directly because it is `protected` and the inheritance is `public`. If `m_value` were `private`, `Derived` would need `getValue()` instead.

**Why public inheritance dominates.** Public inheritance preserves the base interface, which is essential for the "is-a" relationship. Protected and private inheritance occasionally reuse implementation without exposing the base interface, but these cases are rare. Default to `public`.

**When to use protected members.** Prefer `private` members with accessors. Use `protected` data sparingly -- only when derived classes need direct access and the invariant can still be maintained. Overusing `protected` couples the hierarchy tightly, making it harder to refactor.
