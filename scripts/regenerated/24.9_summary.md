So far every derived class you have written inherits from a single base class. C++ also allows **multiple inheritance**, where a class derives from two or more base classes at once. The syntax lists the bases separated by commas:

```cpp
#include <iostream>

class Printable {
public:
    int m_id{};
    void identify() const { std::cout << "id=" << m_id << '\n'; }
};

class Serializable {
public:
    int m_id{};
    void serialize() const { std::cout << "serial=" << m_id << '\n'; }
};

class Widget : public Printable, public Serializable {};

int main() {
    Widget w{};
    w.Printable::m_id = 1;
    w.Serializable::m_id = 2;
    w.identify();   // prints id=1
    w.serialize();  // prints serial=2
}
```

`Widget` contains a complete copy of both `Printable` and `Serializable` -- their members, their functions, everything. Construction proceeds left-to-right: `Printable` is constructed first, then `Serializable`, then `Widget` itself. Destruction is the reverse.

The main complication is **ambiguity**. Both bases above have a member called `m_id`. If you write `w.m_id`, the compiler cannot decide which one you mean and reports an error. You resolve this by qualifying with the base name: `w.Printable::m_id` or `w.Serializable::m_id`. The same disambiguation works for member functions that share a name.

Ambiguity gets worse with the **diamond problem**. Suppose classes `B` and `C` both inherit from `A`, and class `D` inherits from both `B` and `C`. Now `D` contains two separate copies of `A` -- one through `B` and one through `C`. Any access to a member that originated in `A` is ambiguous, and the duplicated state can cause subtle bugs where updating one copy of `A`'s data leaves the other stale.

The diamond problem is not just theoretical -- it appears naturally whenever two independently developed base classes share a common ancestor. C++ has a mechanism to solve it (virtual base classes, covered later), but the diamond is a strong signal that the design should be reconsidered.

The practical guidance is: **avoid multiple inheritance unless you have a clear, compelling reason.** A class that needs behavior from two sources can often use composition instead (hold one as a member rather than inheriting from it). Most production C++ codebases restrict multiple inheritance to narrow cases like inheriting from multiple abstract interfaces, where there are no data members to duplicate. If you do use it, keep base classes small, avoid overlapping member names, and be aware that adding a shared ancestor later creates a diamond.
