A **container class** is a class designed to hold and manage a collection of other objects (called **elements**). You use containers constantly in real programs -- any time you need a list of enemies, a stack of undo operations, or a collection of student records, a container is the right abstraction.

Containers come in two flavors:

- **Value containers** hold copies of the objects they store. When the container is destroyed, the elements are destroyed with it. Most containers you will use fall into this category (`std::vector`, `std::array`, and the custom classes below).
- **Reference containers** hold pointers or references to objects that live elsewhere. The container does not manage the lifetime of the referenced objects. An array of `T*` is the simplest reference container.

A good container class uses **composition** -- it owns a dynamically allocated array as a private member and exposes a clean public interface for adding, removing, and accessing elements. The user of the class never touches raw memory.

```cpp
#include <cassert>
#include <iostream>

class IntStack {
    int* m_data{};
    int m_size{};
    int m_capacity{};
public:
    IntStack(int cap) : m_data{new int[cap]{}}, m_capacity{cap} {}
    ~IntStack() { delete[] m_data; }
    void push(int val) { assert(m_size < m_capacity); m_data[m_size++] = val; }
    int pop() { assert(m_size > 0); return m_data[--m_size]; }
    int size() const { return m_size; }
};
```

This `IntStack` demonstrates the key responsibilities of a container: it allocates storage in the constructor, deallocates in the destructor, and provides `push`, `pop`, and `size` operations that keep the internal state consistent. The caller never calls `new` or `delete`.

Note the `assert` guards -- they catch misuse (pushing to a full stack, popping from an empty one) during development. A production container might throw an exception or resize, but assertions are appropriate for learning.

Because `IntStack` manages a resource, the Rule of Three applies. The class above would need a copy constructor and copy-assignment operator (or they should be deleted) to prevent double-delete bugs. For now, the exercises focus on the container interface itself -- resource-safe copying was covered in Chapter 21.

The standard library provides general-purpose containers (`std::vector`, `std::stack`, `std::map`, etc.) that handle all of these concerns. Writing your own teaches you what those containers do under the hood: allocate, track size vs. capacity, and clean up automatically.
