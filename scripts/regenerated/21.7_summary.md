C++ lets you overload all six comparison operators (`==`, `!=`, `<`, `>`, `<=`, `>=`) so your classes work with the same syntax as built-in types. Each operator takes two operands and returns `bool`. You can implement them as member functions (left operand is `*this`) or as friend free functions when you want implicit conversions on both sides.

The classic pre-C++20 approach requires you to write `operator==` and `operator<` with the real logic, then define the other four in terms of those two. This is tedious and error-prone -- six functions for one concept.

```cpp
#include <compare>
#include <iostream>

class IntPair {
    int m_x{};
    int m_y{};
public:
    IntPair(int x, int y) : m_x{x}, m_y{y} {}
    auto operator<=>(const IntPair&) const = default;
};
```

C++20 solves this with the **spaceship operator** `<=>`, formally called the three-way comparison operator. When you default it with `= default`, the compiler generates all six comparison operators for you by comparing each member in declaration order. The return type is typically `std::strong_ordering` for integral members, which distinguishes *less*, *equal*, and *greater*.

If you need custom logic, you can write `operator<=>` yourself. It returns one of three ordering types from `<compare>`: `std::strong_ordering` (substitutability -- equal values are indistinguishable), `std::partial_ordering` (allows unordered results, like floating-point NaN), or `std::weak_ordering` (equivalence without substitutability). A custom `<=>` still auto-generates `<`, `>`, `<=`, and `>=`, but you must separately default or define `operator==` because the compiler will not synthesize it from a user-provided `<=>`.

One important rule: a defaulted `operator<=>` also gives you a defaulted `operator==`, but a user-provided `operator<=>` does not. If you write custom `<=>` logic and also need `==`, explicitly default or implement `operator==`.

Prefer `= default` whenever memberwise comparison is correct. Only write custom comparison logic when your class has members that should be excluded from comparison or requires non-default ordering semantics.
