When two objects hold `std::shared_ptr` references to each other, neither object's reference count ever reaches zero. Both stay alive forever -- a **circular dependency** (also called a reference cycle). This is the most common way `shared_ptr` leaks memory.

Consider two classes, `Person`, each storing a `std::shared_ptr` to a partner. When you write `alice->partner = bob; bob->partner = alice;`, the reference count for each object reaches 2 (one from `main`, one from the partner pointer). When the `shared_ptr` variables in `main` go out of scope, each count drops to 1 -- but never to 0. Neither destructor runs, and memory leaks silently.

**`std::weak_ptr`** breaks the cycle. A `weak_ptr` observes a `shared_ptr`-managed object without incrementing the reference count. If the partner pointer is a `weak_ptr` instead of a `shared_ptr`, the mutual reference does not keep the objects alive. When the last owning `shared_ptr` goes away, the object is destroyed normally.

Because a `weak_ptr` does not own the object, the object might be destroyed while you still hold the `weak_ptr`. Two member functions let you check:

- **`expired()`** returns `true` if the managed object has been destroyed.
- **`lock()`** returns a `std::shared_ptr` to the object if it still exists, or an empty `shared_ptr` if it has been destroyed. Always use `lock()` before accessing the object -- never dereference a `weak_ptr` directly (it has no `operator*` or `operator->`).

```cpp
#include <iostream>
#include <memory>

struct Node {
    int id;
    std::weak_ptr<Node> next;  // weak, not shared -- breaks cycle
    Node(int id) : id{id} {}
    ~Node() { std::cout << "Node " << id << " destroyed\n"; }
};

int main() {
    auto a = std::make_shared<Node>(1);
    auto b = std::make_shared<Node>(2);
    a->next = b;  // weak_ptr assigned from shared_ptr
    b->next = a;
    // Both nodes are destroyed when a and b leave scope
}
```

The design rule is straightforward: in any relationship where two objects point at each other, make one direction a `weak_ptr`. The owning direction keeps a `shared_ptr`; the back-pointer uses `weak_ptr`. This preserves the reference-count cleanup that makes `shared_ptr` useful in the first place.
