Sometimes you need a class object to behave like a built-in type in certain contexts. **Conversion operators** (also called user-defined type conversions) let you define how an object converts to another type. They are member functions whose name is `operator` followed by the target type, they take no parameters, and they have no explicit return type -- the target type *is* the return type.

```cpp
#include <iostream>

class Fraction
{
    int m_num{};
    int m_den{ 1 };
public:
    Fraction(int n, int d = 1) : m_num{n}, m_den{d} {}

    explicit operator double() const
    {
        return static_cast<double>(m_num) / m_den;
    }
};

int main()
{
    Fraction f{ 3, 4 };
    std::cout << static_cast<double>(f); // 0.75
}
```

You can overload conversions to any type: `operator int()`, `operator bool()`, `operator std::string()`, and so on. Each must be a non-static member function with no parameters.

Without the `explicit` keyword a conversion operator is **implicit**, meaning the compiler will silently insert it wherever a conversion is needed -- function arguments, arithmetic expressions, conditionals. This is almost always dangerous. A class with `operator int()` (non-explicit) can accidentally participate in integer arithmetic and comparisons you never intended, producing confusing bugs with no compiler warning.

The fix is to mark conversion operators `explicit`. An explicit conversion operator is only invoked when you write an explicit cast (`static_cast`, C-style cast, or direct initialization with the target type). The one exception is `operator bool()`: even when declared `explicit`, it is still allowed in boolean contexts such as `if`, `while`, and logical operators (`!`, `&&`, `||`). This special rule exists because testing truthiness is so common that requiring a cast every time would be impractical. The standard library relies on this -- `std::optional`, `std::unique_ptr`, and stream objects all define `explicit operator bool()`.

As a rule of thumb: always make conversion operators `explicit` unless you have a strong reason not to. Prefer named functions (like `toInt()` or `toString()`) when the conversion is lossy or surprising; reserve conversion operators for cases where the target type is a natural, expected interpretation of the object.
