Throughout the previous chapters you modeled relationships between objects using composition and aggregation -- one class *has* another as a member. Inheritance introduces a fundamentally different relationship: **"is-a"**. A `Student` *is a* `Person`. A `Cat` *is an* `Animal`. When this relationship genuinely holds, inheritance lets you define a **base class** with shared state and behavior, then create **derived classes** that automatically acquire everything the base provides while adding their own specializations.

## Why inheritance matters

Without inheritance, two classes that share most of their data and logic must duplicate that code. If `Person` has a name and age, and `Student` needs a name, age, *and* a student ID, you would copy every `Person` member into `Student`. Inheritance eliminates this duplication: `Student` inherits from `Person` and adds only what is new.

Inheritance also opens the door to **polymorphism** -- the ability to treat different derived types through a common base interface -- which later chapters will explore in depth.

## Basic syntax

A derived class names its base class after a colon in the class definition. The keyword `public` before the base name is the most common form and preserves the base class's public interface in the derived class.

```cpp
#include <iostream>
#include <string>

class Person {
public:
    std::string name{};
    int age{};
    void greet() const {
        std::cout << name << ", age " << age << '\n';
    }
};

class Student : public Person {
public:
    int studentId{};
};
```

With this definition, a `Student` object has `name`, `age`, and `greet()` from `Person` plus its own `studentId`. No code was copied.

## What the derived class gets

Public inheritance gives the derived class access to every **public** member of the base class as if those members were defined directly in the derived class. Clients of the derived class can use those members transparently. Private members of the base exist inside the derived object's memory layout, but the derived class cannot access them directly -- that boundary is preserved.

## When to use inheritance

Apply inheritance only when the "is-a" test passes in both directions of reasoning: every `Student` truly is a `Person`, and anywhere a `Person` is expected, substituting a `Student` should make sense. If the relationship is "has-a" (a `Car` *has an* `Engine`), composition remains the right tool. Misusing inheritance where composition fits leads to rigid, fragile hierarchies.
