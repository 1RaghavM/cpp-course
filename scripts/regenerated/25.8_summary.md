In lesson 24.9 you saw the diamond problem: when `B` and `C` both inherit from `A`, and `D` inherits from both `B` and `C`, the object `D` contains **two separate copies** of `A`. Any access to `A`'s members is ambiguous, and the duplicated state leads to bugs. C++ solves this with **virtual base classes**.

The fix is to add the keyword `virtual` to the inheritance declaration in the intermediate classes:

```cpp
class A {
public:
    int m_value{};
    A(int v) : m_value{v} {}
};
class B : virtual public A {  // virtual inheritance
public:
    B(int v) : A{v} {}
};
class C : virtual public A {  // virtual inheritance
public:
    C(int v) : A{v} {}
};
class D : public B, public C {
public:
    D(int v) : A{v}, B{v}, C{v} {}  // D inits A directly
};
```

With `virtual` inheritance, `D` now contains exactly **one shared copy** of `A`, no matter how many paths lead to it. The ambiguity disappears -- `D` can access `m_value` directly because there is only one.

This introduces an important **constructor responsibility rule**: when a class has a virtual base, the **most-derived class** must directly initialize that virtual base in its member initializer list. In the example above, `D`'s constructor calls `A{v}` explicitly. Even though `B` and `C` also list `A` in their initializer lists, those calls are **ignored** when constructing a `D` -- only `D`'s direct call to `A`'s constructor is used. This prevents conflicting initialization from different intermediate classes.

If the most-derived class does not explicitly initialize the virtual base, the virtual base's **default constructor** is called. This is a common source of confusion: you add a parameter to `A`'s constructor, `B` and `C` pass it correctly, but `D` forgets to initialize `A` directly, so the compiler looks for `A()` and fails.

Construction order with virtual bases is also special: virtual base classes are always constructed **before** any non-virtual base classes, regardless of where they appear in the inheritance list. Destruction is the reverse.

Virtual base classes solve a real problem, but they add complexity. Use them only when the diamond genuinely exists and a single shared base is the correct semantic. In most codebases, prefer composition or pure abstract interfaces to avoid diamonds entirely.
