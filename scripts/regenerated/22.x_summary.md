## Chapter 22 Summary

This chapter addressed the two biggest pain points of manual memory management in C++ -- ownership ambiguity and expensive copies -- by introducing move semantics and smart pointers.

### Ownership and the case for smart pointers (22.1)

Raw pointers carry no ownership information. When multiple pointers alias the same heap object, it is unclear which call site should `delete` it, leading to leaks, double-frees, and dangling pointers. Smart pointers solve this by encoding ownership in the type system and calling `delete` automatically in their destructors.

### R-value references (22.2)

Every expression is either an l-value (has a persistent address) or an r-value (temporary). An r-value reference (`T&&`) binds only to r-values, extending the temporary's lifetime. Overloading on `T&` vs `T&&` lets the compiler route temporaries to a move path instead of a copy path -- the foundation of move semantics.

### Move constructors and move assignment (22.3)

A move constructor (`T(T&& src) noexcept`) transfers ownership by copying the source's internal pointer and nullifying it. Move assignment does the same after releasing the target's existing resource. After a move, the source is in a valid but unspecified state.

### std::move (22.4)

`std::move` (from `<utility>`) casts an l-value to an r-value reference, enabling the compiler to select a move overload. It does not move anything by itself -- it simply signals you no longer need the object's value.

### std::unique_ptr (22.5)

`std::unique_ptr<T>` models exclusive ownership. It cannot be copied, only moved. Create one with `std::make_unique<T>(args...)`. When destroyed or reassigned, the managed object is deleted. Passing by value transfers ownership; passing by `const` reference allows observation without transfer.

### std::shared_ptr (22.6)

`std::shared_ptr<T>` models shared ownership via reference counting. The object is deleted when the last owning `shared_ptr` is destroyed. Create with `std::make_shared<T>(args...)`, which performs a single allocation for both the control block and the object. Use `use_count()` to inspect the reference count.

### Circular dependencies and std::weak_ptr (22.7)

When two objects hold `shared_ptr` references to each other, neither count reaches zero, causing a leak. `std::weak_ptr<T>` breaks the cycle by observing without incrementing the count. Call `lock()` to obtain a `shared_ptr` that is non-null only if the object still exists, or `expired()` to check validity without locking.
