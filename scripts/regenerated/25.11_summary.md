You already know how to overload `operator<<` as a friend or free function (Ch 21). You also know that `virtual` lets derived classes override behavior through base pointers and references. But you cannot make `operator<<` virtual directly, because virtual dispatch works on member functions called on the left-hand operand, and the left-hand operand of `operator<<` is `std::ostream` -- not your class.

The solution is a two-layer pattern. First, define a `virtual` member function -- conventionally named `print` -- that takes a `std::ostream&` and writes the object's data to it. Each derived class overrides `print` to output its own representation. Second, define a single non-member `operator<<` in the base class that delegates to `print`:

```cpp
#include <iostream>
class Base {
public:
    virtual void print(std::ostream& out) const { out << "Base"; }
    friend std::ostream& operator<<(std::ostream& out, const Base& b) {
        b.print(out);
        return out;
    }
    virtual ~Base() = default;
};
class Derived : public Base {
    int m_x{};
public:
    Derived(int x) : m_x{x} {}
    void print(std::ostream& out) const override { out << "Derived(" << m_x << ")"; }
};
```

Now `std::cout << derived` works polymorphically. The single `operator<<` defined in `Base` calls `print()` on whichever object is actually referenced. Because `print` is virtual, the call dispatches to the correct override at runtime. You never need to write another `operator<<` in any derived class.

This pattern has three moving parts: (1) a virtual `print` member in the base, (2) an `override` of `print` in each derived class, and (3) exactly one `operator<<` friend in the base that calls `print`. If you add a new derived class, you only add a `print` override -- the streaming operator just works.

Always mark the base class destructor `virtual` when using this pattern, since objects will typically be used through base pointers or references. Use `override` on every derived `print` to catch signature mismatches at compile time.
