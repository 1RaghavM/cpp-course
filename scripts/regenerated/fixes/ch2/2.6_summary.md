## Why Functions Are Useful, and How to Use Them Effectively

In "Introduction to functions" you saw how to define and call a function. Now let's talk about *when* to write one and how to keep functions clean as your programs grow.

### The three reasons to extract a function

1. **Avoid duplicate code.** If you copy-paste logic, bugs get fixed in one place and silently survive in the other. A function lets you write the logic once and call it anywhere.
2. **Name a concept.** A function called `computeArea` communicates intent instantly; two lines of arithmetic scattered inline do not.
3. **Limit scope.** Recall from "Introduction to local scope" that every variable lives only inside its enclosing `{}`. Pushing a self-contained task into its own function keeps its variables from polluting the caller.

### Single Responsibility Principle

Each function should do **one thing**. A function that calculates a value *and* prints it is doing two things -- split it. This makes each piece easier to test, reuse, and understand.

### Return values carry the result -- don't mix concerns

From "Function return values (value-returning functions)" you know a function returns exactly one value. As seen in "Void functions", if a function only performs a side effect (like printing), make it `void`. If it computes something, make it return that value and leave printing to the caller:

```cpp
#include <iostream>

// computes only -- no printing
int cubeVolume(int side) {
    return side * side * side;
}

int main() {
    int vol { cubeVolume(3) };
    std::cout << "Volume: " << vol << '\n';
    return 0;
}
```

### Parameters keep functions general

"Introduction to function parameters and arguments" showed that parameters let a function work on different data each call. Prefer parameters over hard-coded values so the function is reusable without modification.

### When *not* to extract a function

Don't extract a function just to shorten `main`. One-liners used only once are fine inline. The signal to extract is **reuse or clarity**, not line count.

### Practical checklist
- Does the function have a single, nameable job? Good.
- Is logic duplicated elsewhere? Extract it.
- Does it mix computation with I/O? Split it.
- Are parameters named for what they represent, not their type? Good.

Following these habits from the start keeps C++ programs readable as they scale from dozens to thousands of lines.
