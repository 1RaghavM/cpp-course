## How to Design Your First Programs

Writing code is easy. Knowing *what* to write before you type a single line is the skill that separates working programs from frustrating debugging sessions. This lesson is about the design step that comes before the code.

### Start with the problem, not the syntax

When you get a new problem, resist the urge to open your editor. Instead, write down -- in plain English -- what your program must do. Break that into a list of steps. Each step that is non-trivial and reusable is a candidate for its own function (see "Introduction to functions" and "Why functions are useful, and how to use them effectively").

This approach is called **top-down design**: start from the high-level goal, decompose into sub-tasks, and implement each sub-task as a focused function.

### A concrete process

1. **Clarify outputs** -- what does the program print or return?
2. **Identify inputs** -- what data does it need, and where does it come from?
3. **List the steps** -- write pseudocode if helpful.
4. **Map steps to functions** -- each cohesive step becomes a function with a clear name and a single responsibility.
5. **Write `main` last** -- `main` should read like a summary of your pseudocode, calling the functions you defined.

### Example: score report

```cpp
#include <iostream>

int computeTotal(int a, int b, int c) {
    return a + b + c;
}

void printReport(int total, int count) {
    std::cout << "Total: " << total << '\n';
    std::cout << "Average: " << total / count << '\n';
}

int main() {
    int s1{}, s2{}, s3{};
    std::cin >> s1 >> s2 >> s3;
    int total { computeTotal(s1, s2, s3) };
    printReport(total, 3);
    return 0;
}
```

`main` is a four-line story: read, compute, report. Each piece lives in its own function with a name that explains its purpose.

### Common pitfalls

- **Monolithic `main`:** stuffing all logic into `main` makes code hard to test and reason about. Use forward declarations (see "Forward declarations and definitions") so you can define helpers above `main` or in separate files (see "Programs with multiple code files").
- **Naming too late:** choose clear function and variable names *while designing*, not after. The name forces you to articulate what the function actually does.
- **Over-engineering:** do not add functions for their own sake. A two-line computation used once can stay inline.

### Design is iterative

Your first design will not be perfect. That is expected. Write the simplest version that works, then refactor by extracting repeated logic into functions. Each iteration, your program becomes easier to read and extend.
