The **parenthesis operator** `operator()` lets an object be called as if it were a function. A class that overloads `operator()` is called a **functor** (or function object). Unlike most operators, `operator()` can accept any number of parameters and return any type.

A functor carries state in its member variables while still being callable. This makes functors more flexible than plain function pointers, which cannot hold state without resorting to global variables.

```cpp
#include <algorithm>
#include <iostream>
#include <vector>

struct GreaterThan {
    int threshold;
    bool operator()(int value) const {
        return value > threshold;
    }
};

int main() {
    std::vector v{ 8, 2, 5, 1, 9, 3 };
    auto count{ std::count_if(v.begin(), v.end(),
                              GreaterThan{ 4 }) };
    std::cout << count << '\n'; // 3
}
```

`GreaterThan{ 4 }` constructs a temporary functor whose `threshold` is 4. When `std::count_if` calls the functor for each element, `operator()` compares `value` against that stored threshold. The same class can be reused with different thresholds without writing separate functions.

Because `operator()` is a regular member function, it can be overloaded with different parameter lists. A single class could define `operator()(int)` and `operator()(int, int)` to behave differently depending on how many arguments the caller passes.

Functors work anywhere a callable is expected. Standard algorithms like `std::sort`, `std::count_if`, and `std::transform` all accept functors. Before lambdas were introduced in C++11, functors were the primary way to pass custom behavior to algorithms. Lambdas are actually implemented by the compiler as anonymous functor classes behind the scenes -- a lambda with captures becomes a class whose members store the captured values and whose `operator()` contains the lambda body.

Prefer lambdas for short, one-off callables since they are more concise. Use an explicit functor class when you need a reusable callable with named state, when the logic is complex enough to deserve a descriptive class name, or when the same callable type must be shared across multiple translation units.
