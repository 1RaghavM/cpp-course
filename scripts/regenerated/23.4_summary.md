An **association** models a "uses-a" relationship between two otherwise unrelated objects. Unlike composition and aggregation, neither object is part of the other and neither manages the other's lifetime. Both objects exist independently and can function without each other. A Doctor "uses" a Patient and vice-versa, but destroying one does not affect the other.

Associations can be **unidirectional** (only one side knows about the other) or **bidirectional** (both sides hold references to each other). They are typically implemented with **pointers** rather than references, because an association may be optional (null when no relationship exists) or changeable over time.

```cpp
#include <iostream>
#include <string>
#include <vector>

class Patient;

class Doctor {
    std::string m_name;
    std::vector<Patient*> m_patients{};
public:
    Doctor(std::string name) : m_name{name} {}
    void addPatient(Patient* p) { m_patients.push_back(p); }
    const std::string& getName() const { return m_name; }
    const std::vector<Patient*>& getPatients() const { return m_patients; }
};
```

Here `Doctor` holds a vector of non-owning pointers to `Patient` objects. The doctor does not create or destroy patients — it merely knows about them. The patient objects are created and destroyed elsewhere, and the pointers simply go stale if a patient is removed without updating the doctor.

Because neither side owns the other, you must be careful about **dangling pointers**. If a patient is destroyed while a doctor still points to it, using that pointer is undefined behavior. In practice you manage this by explicitly removing the association (clearing the pointer or erasing it from the vector) before destroying either object.

**Association vs. aggregation:** aggregation implies a whole/part hierarchy — a department *has* teachers. Association implies a peer relationship — a doctor *uses* a patient. If you can phrase the relationship as "X uses Y" or "X works with Y" rather than "X has Y," it is an association.

To summarize the key properties: the associated objects are otherwise unrelated, neither is part of the other, neither manages the other's lifetime, the relationship can be unidirectional or bidirectional, and it is implemented through pointers since the link may need to be optional or reassigned.
